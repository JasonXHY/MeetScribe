## 二、历史讨论记录整合

### 2.1 打包方案（DEVPLAN_V2, 2026-05-29）

- **打包工具**：PyInstaller 目录模式
- **模型策略**：首次启动云端下载（安装包不含模型，约 2.5 GB）
- **安装器**：NSIS 安装包 + 便携版 ZIP 双轨分发
- **代码签名**：EV 代码签名证书（约 $500/年）
- **最大障碍**：杀毒软件误报（PyInstaller 生成的 exe 误报率极高）
- **实施条件**：功能完善且稳定后再考虑

**打包前待验证问题**：
1. 命令行窗口闪现：Windows 上 `multiprocessing.Process`（spawn 模式）启动子进程时可能触发短暂控制台窗口。打包时需用 `--noconsole`（`--windowed`）参数验证；若仍有闪现，需在子进程入口设置 `STARTUPINFO.dwFlags |= STARTF_USESHOWWINDOW`
2. FunASR/PyTorch 内部的 subprocess 调用点需排查

### 2.2 Nuitka 备选方案评估（ERROR_TRACKING, 2026-05-29）

- 将 Python 编译为 C 再编译为原生 exe
- 运行时仍依赖 Python 解释器和 GIL → 对"未响应"问题无帮助
- 对 PyTorch 推理无提速（瓶颈在 C++/CUDA）
- 内存节省仅几十 MB
- **结论：对 MeetScribe 场景收益很小，不采用**

### 2.3 架构演进规划（ERROR_TRACKING, 2026-05-29）

**当前架构瓶颈**：
- 打包困难（6-8 GB 总大小）
- 杀毒误报率高
- 启动慢、内存占用大

**远期演进方向：混合架构**
```
原生前端（C# WPF / WinUI 3）  ←→  Python 后端（模型推理）
     ↑                                   ↑
  UI 渲染/交互                      FunASR/PyTorch
  打包简单（单文件 exe）             AI 生态完整
  响应快                             进程隔离
通信方式：Socket / gRPC / HTTP
```

**各方案对比**：

| 方案 | 运行时 | 内存基线 | 打包难度 | AI 生态 | 适合场景 |
|------|--------|---------|---------|---------|---------|
| Python + customtkinter | 解释器 + GIL | 50-80MB + 模型 | 极难 | 无敌 | AI 原型 |
| C# WPF (.NET) | CLR + JIT | 30-50MB | 简单（单文件） | 弱 | Windows 桌面标准 |
| C++ / Qt | 原生机器码 | 最小 | 中等 | 弱 | 性能敏感工具 |
| Electron (JS/TS) | V8 + Chromium | 150-300MB | 中等 | 弱 | 跨平台/内容展示 |

**行业参考**：Adobe（C++ 前端 + Python 脚本）、DaVinci Resolve（C++ 核心 + Python/Lua）、Whisper Desktop（原生 UI + Python/ONNX）

**实施优先级**：
1. 当前阶段：完善 Python 版本功能
2. 中期：功能稳定后做 PyInstaller 打包分发（便携版）
3. 远期：如需大规模分发，考虑混合架构重写

## 三、文件复用策略细节（待讨论）

### 3.1 新项目目录结构（初步构想）

```
C:\侧耳倾听\                    ← 正式版项目目录
  src\                           ← 重构后的源代码
  config\                        ← 配置文件
  data\                          ← 音色库数据
  assets\                        ← 新设计的图标资源
  tests\                         ← 测试代码
  docs\                          ← 文档
  侧耳倾听.bat                   ← 启动脚本
  侧耳倾听.lnk                   ← 快捷方式

C:\MeetScribe\                   ← 开发目录（保留不动）
  models_cache\                  ← 模型文件（~1.8GB，不复制）
  recordings\                    ← 录音文件（用户数据）
  transcripts\                   ← 转写输出（用户数据）
```

### 3.2 模型文件复用方案（待确定）

**方案 A：共享目录引用**
- 正式版代码中的模型路径指向 `C:\MeetScribe\models_cache\`
- 优点：零额外存储
- 缺点：两个项目耦合，删除开发目录会影响正式版

**方案 B：符号链接 / Junction**
- 在 `C:\侧耳倾听\models_cache\` 创建指向 `C:\MeetScribe\models_cache\` 的目录联接（`mklink /J`）
- 优点：代码中模型路径不变，存储不重复
- 缺点：需要管理员权限创建，用户可能不理解

**方案 C：配置化模型路径**
- 在 `config/settings.json` 中增加 `model_cache_dir` 配置项
- 正式版首次启动时让用户选择模型目录（默认 `C:\MeetScribe\models_cache\`）
- 优点：最灵活，解耦
- 缺点：首次体验多一步配置

### 3.3 可复用工具

以下工具/资源在两个项目间可共享，不需要复制：
- FunASR 模型文件（~1.8GB）
- Node.js 依赖（docx 库）
- Python 虚拟环境（如果使用 venv）
- VB-Audio Cable 驱动（系统级安装）
