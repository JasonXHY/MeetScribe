## 一、产品概述

### 1.1 这是什么

**侧耳倾听**是一款 Windows 桌面应用，核心功能是**将会议录音自动转化为结构化的会议纪要**。

一句话描述：**录音 → 转文字 → 识别谁在说话 → AI 生成会议纪要**，全程只需点两下按钮。

### 1.2 解决什么问题

开会时，参会者面临三个痛点：

1. **记录成本高**：手动记笔记会分散注意力，且无法完整记录所有发言
2. **整理耗时**：会后需要反复听录音才能整理出会议纪要，1小时会议可能需要2-3小时整理
3. **发言归属难**：多人讨论时，很难区分每句话是谁说的

侧耳倾听通过 AI 技术自动完成"听录音→转文字→认人→总结"的全流程，将会议记录的整理时间从小时级降到分钟级。

### 1.3 目标用户

| 用户类型 | 典型场景 | 核心需求 |
|----------|----------|----------|
| 企业员工 | 参加周会、项目讨论、客户会议 | 快速生成会议纪要，减少手动整理 |
| 项目经理 | 管理多个项目会议 | 会后快速回顾决策事项和待办 |
| 团队负责人 | 主持团队讨论 | 了解每个成员的发言内容 |
| 个人用户 | 学习培训、讲座记录 | 将录音转化为可搜索的文字 |

### 1.4 核心价值

- **省时**：1小时会议录音，5分钟内生成完整纪要
- **准确**：AI 说话人分离 + 声纹识别，准确区分每个发言者
- **智能**：自动提取讨论要点、决策事项、待办事项
- **灵活**：支持多种输出格式（Markdown/纯文本/SRT字幕/JSON）
- **隐私**：核心转写在本地完成，只有 AI 摘要使用云端 API

## 二、业务场景详解

### 场景一：线下现场会议

**场景描述**：用户在会议室中，使用笔记本电脑录制现场多人讨论。

**用户操作流程**：
1. 打开应用 → 选择「现场会议」模式（麦克风录音）
2. 点击「开始录音」→ 应用通过笔记本内置麦克风采集音频
3. 会议进行中 → 应用实时显示录音时长和状态
4. 会议结束 → 点击「停止录音」→ WAV 文件自动保存
5. 点击「开始转写」→ 后台启动 AI 转写（SenseVoice ASR + CAM++ 说话人分离）
6. 转写完成 → 自动进行 AI 摘要生成（云端大模型）
7. 用户查看转写结果 → 可导出为 Markdown/纯文本/SRT字幕/JSON
8. 用户管理发言人 → 通过「发言人管理」弹窗为 Speaker 1/2/3 命名

**为什么这样设计**：
- 麦克风单轨录音是最简单的场景，适合线下会议
- 自动保存避免用户忘记保存导致录音丢失
- 转写完成后自动触发摘要，减少用户操作步骤

**验收标准**：
- [ ] REC-001: 能正确录制 16kHz 单声道 WAV 文件
- [ ] REC-002: 录音过程中实时显示时长（每秒更新）
- [ ] TRN-001: 转写结果包含时间戳和说话人标识
- [ ] AI-001: AI 摘要包含参会人员、讨论要点、决策事项、待办事项

### 场景二：线上远程会议（核心差异化功能）

**场景描述**：用户在电脑上参加线上会议（腾讯会议/飞书/Zoom等），需要录制电脑播放的声音（远端参会者发言）+ 用户自己的麦克风声音。

**用户操作流程**：
1. 打开应用 → 选择「线上会议」模式（双轨录音）
2. 安装 VB-Audio Cable 虚拟音频设备（首次使用时引导）
3. 在会议软件中将音频输出设置为 VB-Audio Cable
4. 点击「开始录音」→ 应用同时录制：
   - 麦克风轨：用户自己说的话
   - 系统音频轨：通过 VB-Audio Cable/WASAPI Loopback 录制远端参会者声音
5. 会议结束 → 两个 WAV 文件自动保存（`meeting_xxx.wav` + `meeting_xxx_sys.wav`）
6. 点击「开始转写」→ 两轨分别转写 → 自动按时间戳合并
7. 合并结果中自动标注「本地-N」（用户自己）和「远程-N」（远端参会者）

**为什么这样设计**：
- 双轨录音是核心差异化功能，解决线上会议录音的最大痛点
- 分开录制再合并，比混音录制更准确（可以区分本地/远程）
- VB-Audio Cable 是免费的虚拟音频设备，兼容性好
- WASAPI Loopback 作为回退方案，无需安装额外软件

**验收标准**：
- [ ] REC-005: 能同时录制麦克风和系统音频两个轨道
- [ ] REC-006: 系统音频捕获优先 VB-Audio Cable，回退 WASAPI Loopback
- [ ] TRN-011: 双轨文件能正确合并，按时间戳排序
- [ ] SPK-008: 合并结果中正确标注「本地-N」和「远程-N」

### 场景三：已有录音文件转写

**场景描述**：用户已有录音文件（可能是其他设备录制的），需要转写为文字。

**用户操作流程**：
1. 点击「添加文件」→ 选择一个或多个音频文件
2. 支持格式：WAV/MP3/FLAC/M4A/AAC/OGG/WMA
3. 文件列表显示：文件名、主题、时长、大小、状态
4. 选中文件 → 点击「开始转写」→ 后台处理
5. 多文件可批量转写或合并转写

**为什么这样设计**：
- 支持多种音频格式，方便用户导入不同来源的录音
- 批量转写提高效率，适合有大量录音需要处理的用户
- 合并转写适合将多段连续录音合并为一场会议处理

**验收标准**：
- [ ] FILE-001: 能正确导入 WAV/MP3/FLAC/M4A/AAC/OGG/WMA 格式
- [ ] TRN-002: 批量转写时按队列顺序处理
- [ ] TRN-003: 合并转写时按说话人合并多段录音

### 场景四：音色库管理（声纹识别）

**场景描述**：用户需要识别会议中的已知发言人的身份。

**用户操作流程**：
1. 进入「音色库」页面
2. 点击「添加音色」→ 打开录音弹窗
3. 按预设文本朗读 → 应用提取声纹嵌入向量
4. 保存到音色库（带质量评分）
5. 下次转写时，自动匹配音色库 → 已知发言人自动命名

**为什么这样设计**：
- 预设文本朗读可以确保声纹提取质量（文本覆盖音素范围）
- 质量评分帮助用户了解声纹数据的可靠性
- 自动匹配减少手动命名的工作量
- FIFO 淘汰控制存储大小，避免声纹库无限膨胀

**验收标准**：
- [ ] VPR-002: 录音朗读后能正确提取声纹嵌入向量
- [ ] VPR-006: 转写时自动匹配音色库，匹配结果准确
- [ ] VPR-007: 质量评分基于片段数和时长计算
- [ ] VPR-009: FIFO 淘汰每说话人最多保留5个样本

### 场景五：AI 智能摘要

**场景描述**：转写完成后，自动生成结构化的会议纪要。

**AI 摘要输出结构**：
```markdown
# 会议纪要

## 参会人员
- 张三、李四、王五

## 讨论要点
1. Q3 销售目标讨论...
2. 新产品发布计划...

## 决策事项
- 决定 Q3 目标为 500 万
- 新产品 8 月 1 日发布

## 待办事项
- 张三：整理销售数据
- 李四：准备产品文档
```

**为什么这样设计**：
- 结构化输出便于用户快速获取关键信息
- 参会人员列表帮助确认出席情况
- 讨论要点便于回顾会议内容
- 决策事项和待办事项便于后续跟进

**验收标准**：
- [ ] AI-001: 摘要包含参会人员、讨论要点、决策事项、待办事项
- [ ] AI-002: LLM 纠错能修正 ASR 错字、删除乱码 token
- [ ] AI-003: 发言人姓名提取支持正则 + LLM 兜底

### 5.1 完整数据流

```
用户操作
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│  录音阶段                                                     │
│  UnifiedRecorder → WAV文件(16kHz单声道)                        │
│  - mic模式: 仅麦克风                                          │
│  - dual模式: 麦克风 + 系统音频(两个WAV文件)                     │
└─────────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│  转写阶段（子进程）                                            │
│  Transcriber.transcribe_staged() 4阶段流水线:                  │
│  1. SenseVoice ASR + fsmn-vad 分段                            │
│  2. CAM++ 说话人分离 + 按人嵌入提取                             │
│  3. ct-punc 标点恢复                                          │
│  4. 后处理(清理/合并碎片)                                      │
│  → 格式化输出(llm-md/md/txt/srt/json/html/csv/vtt)            │
└─────────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│  声纹匹配阶段                                                  │
│  VoiceprintLibrary.match_with_confidence()                   │
│  → 已知发言人自动命名                                          │
│  → 未知发言人保持 Speaker N 标识                               │
└─────────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│  AI 处理阶段（可选）                                           │
│  1. SpeakerNamer: 正则+LLM 提取发言人姓名                      │
│  2. AIService.generate_correction: LLM 纠错                   │
│  3. AIService.generate_summary: AI 摘要生成                    │
│  → 输出结构化会议纪要                                          │
└─────────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│  持久化阶段                                                    │
│  FileManager: 文件列表 + 历史记录                               │
│  TranscriptionQueue: 转写任务队列                               │
│  VoiceprintLibrary: 声纹嵌入向量持久化                          │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 进程间通信

```
┌──────────────────────┐          ┌──────────────────────┐
│  主进程 (GUI)          │          │  子进程 (转写)          │
│                      │          │                      │
│  TranscriptionHandler│          │  TranscribeWorker    │
│  - start()           │ ──────→  │  - 读取音频文件        │
│  - _poll()           │ ←──────  │  - 执行转写           │
│  - _process_message()│          │  - 发送消息到Queue     │
│                      │          │                      │
│  multiprocessing.Queue│ ←─────→ │  multiprocessing.Queue│
│  (自适应轮询 50-500ms) │          │                      │
└──────────────────────┘          └──────────────────────┘

消息协议：
- ("status", "转写中...")        → 状态更新
- ("log", "正在处理...")         → 日志输出
- ("processing", file_id)        → 文件状态更新
- ("file_done", path, spk_data)  → 转写完成
- ("merge_done", paths)          → 合并完成
- ("error", msg)                 → 错误处理
- ("done",)                      → 全部完成
```

### 6.2 交互设计原则

1. **最少点击原则**：核心操作（录音→转写→查看结果）不超过3次点击
2. **状态可见原则**：所有操作都有明确的状态反馈（录音中、转写中、完成）
3. **错误恢复原则**：错误发生时提供重试或撤销选项
4. **一致性原则**：相同功能使用相同的交互模式
5. **容错原则**：危险操作（删除、清空）需要确认

## 七、技术约束

### 7.1 必须使用的技术

| 技术 | 版本 | 用途 | 不可替换原因 |
|------|------|------|-------------|
| Python | 3.10+ | 开发语言 | 生态丰富，AI库支持好 |
| PySide6 | 6.x | GUI框架 | Qt6 绑定，跨平台，硬件加速 |
| FunASR | latest | ASR引擎 | 阿里出品，中文优化，含SenseVoice+CAM++ |
| PyAudioWPatch | latest | 音频录制 | 支持 WASAPI Loopback（Windows独有） |
| MiMo API | v1 | AI摘要/纠错 | 小米大模型，中文效果好 |

### 7.2 可选技术

| 技术 | 用途 | 替代方案 |
|------|------|----------|
| VB-Audio Cable | 虚拟音频设备 | WASAPI Loopback（自动回退） |
| Ollama | 本地LLM | 云端API（需联网） |
| FFmpeg | 音频处理 | 已集成到FunASR |

### 7.3 不能做的事

1. **不能使用 GPL 协议的库**：项目使用 Unlicense，不能引入 GPL 依赖
2. **不能硬编码 API Key**：必须通过配置文件管理，配置文件不能提交到 Git
3. **不能在主线程执行耗时操作**：录音、转写、AI处理必须在子线程/子进程
4. **不能直接操作跨线程 UI**：必须通过 Qt Signal/Slot 机制
5. **不能假设用户有 GPU**：必须支持纯 CPU 运行

5. **声纹识别准确率有限**：嘈杂环境或短音频可能识别不准
