### BUG-01：录音回调未注册 — 录音后文件不会出现在列表中

**文件**：`app.py` 第 107-114 行
**现象**：`UnifiedRecorder` 创建后，三个回调方法（`_on_recorder_state_change`、`_on_recorder_save`、`_on_recorder_stop_complete`）已定义（第 239-278 行），但从未赋值给 `recorder` 实例。

**影响**：
1. 录音状态变化时 GUI 不会更新（计时器不启动、按钮状态不变）
2. 录音保存后文件不会自动添加到 `file_manager`
3. 录音停止后不会触发双轨合并、不会刷新文件列表
4. 录音过程中 `_recording` 和 `_paused` 状态变量始终为初始值

**修复方案**：在 `app.py` 第 114 行（`UnifiedRecorder initialized` 之后）添加：

```python
self.recorder.on_state_change = self._on_recorder_state_change
self.recorder.on_save = self._on_recorder_save
self.recorder.on_stop_complete = self._on_recorder_stop_complete
```

**验证方式**：开始录音 → 观察计时器是否走动 → 停止录音 → 观察文件列表是否自动新增录音文件

## 三、中等问题（P1 — 影响部分功能或用户体验）

### BUG-08：`_transcribe_single` 中 `add_to_queue` 和 `start` 重复调用

**文件**：`home_page.py` 第 556-557 行
**现象**：先调用 `handler.add_to_queue([file_path])` 将文件加入队列，再调用 `handler.start([file_path], fmt, {}, "")`。`start` 内部也会操作队列，可能导致同一文件在队列中出现两次。

**对比原版**：原版 `_transcribe_single` 只调用 `self._handler.start([file_path], fmt, speaker_names, out_dir)`，不预先 `add_to_queue`。

### BUG-09：`file_list_view.py` 中 `lstrip` 用于前缀剥离

**文件**：`file_list_view.py` 第 448 行
**现象**：`current.lstrip("● ").lstrip("📎 ")` 使用 `lstrip` 进行前缀剥离。`lstrip` 是按字符集合逐字符剥离，而非按字符串前缀匹配。例如 `"● 📎 test"` 会被过度剥离。

**修复方案**：改用条件前缀剥离：

```python
if current.startswith("● "):
    current = current[2:]
if current.startswith("📎 "):
    current = current[2:]
```

### BUG-10：`multiprocessing.Queue.empty()` 在多进程环境不可靠

**文件**：`transcription.py` 第 153 行
**现象**：`self._queue.empty()` 在多进程场景下有竞态条件。Python 文档明确指出 `Queue.empty()` 在多进程中不可靠。

**修复方案**：改用 `get_nowait()` + `queue.Empty` 异常处理：

```python
while True:
    try:
        msg = self._queue.get_nowait()
        self._process_message(msg)
    except queue.Empty:
        break
```

### BUG-11：`_on_done` 可能被重复调用

**文件**：`transcription.py`
**现象**：`_poll` 方法中同时检查进程结束（`process.is_alive()` 为 False）和队列中的 `"done"` 消息。如果两者在同一轮 poll 中被检测到，`_on_done` 可能被调用两次，导致 `transcription_done` 信号重复发射。

**修复方案**：添加防重入标记 `self._done_called`，在 `_on_done` 入口检查并设置。

### BUG-12：`_speaker_embeddings` 和 `_sentences` 在新任务时未重置

## 五、UI / 样式问题

### 5.1 QSS 样式大量内联

几乎每个组件都在代码中写内联 `setStyleSheet()` 字符串，而非使用 `styles.py` 中定义的 `MAIN_STYLESHEET`。这导致：
- 样式分散在 12 个文件中，修改一个颜色需要搜索所有文件
- 重复代码多（如录音栏三种状态各有一套完整的样式字符串）
- 后续主题切换（如暗色模式）工作量巨大

**建议**：将常用样式提取到 `styles.py` 作为常量，或使用 QSS 类选择器（`setProperty("cssClass", "xxx")`）。

### 5.2 主按钮颜色使用红色

`styles.py` 第 164 行 `QPushButton[cssClass="primary"]` 的 `background-color` 使用了 `C_ERROR`（红色 #C42B1C）。通常主按钮应使用 `C_ACCENT`（蓝色）。虽然当前代码中 `cssClass="primary"` 未被实际使用，但如果后续启用会造成困惑。

### 5.3 硬编码颜色值

QSS 中存在硬编码颜色（如 `#A52318`、`#FDE8E8`、`#C0C0C0`），未提取为常量。

## 六、代码质量问题

### 6.1 VoiceprintLibrary 每次操作重新实例化

**文件**：`voiceprint_page.py`
**现象**：`refresh_list`、`_show_speaker_detail`、`_add_speaker` 等方法每次都 `from voiceprint import VoiceprintLibrary; library = VoiceprintLibrary()`。如果 `VoiceprintLibrary` 初始化需要扫描磁盘文件，这会造成性能问题。

**建议**：在 `VoiceprintPage.__init__` 中缓存实例。

### 6.2 SRT 转换时间计算过于简单

**文件**：`dialogs.py` 第 265-274 行
**现象**：每条字幕的结束时间固定为开始时间 +3 秒，可能导致字幕时间重叠。

**建议**：使用下一条字幕的开始时间作为当前字幕的结束时间（最后一条用 +3s 兜底）。

### 6.3 Config.set 默认立即写盘

**文件**：`config.py`
**现象**：`set(key, value, save=True)` 默认每次调用都触发原子写入。在 `_on_save` 中连续调用多次 `set` 会导致多次磁盘 IO。

**建议**：批量操作时传入 `save=False`，最后手动调用 `save()`。

### 6.4 日志文件无轮转机制

**文件**：`main.py`
**现象**：日志写入 `meetscribe.log`，无大小限制和轮转，长期使用可能导致文件过大。

**建议**：使用 `RotatingFileHandler` 替代 `FileHandler`。

### 6.5 模块间耦合度高

`HomePage` 直接操作 `self._app.recorder`、`self._app.file_manager`、`self._app._transcription_handler`。`VoiceprintPage` 也直接访问 `self._app._home_page._log()`。紧耦合使单元测试困难，且任何一个组件的变更可能影响其他组件。

**建议**：通过 Signal/Slot 解耦，而非直接访问其他组件的私有属性。

## 七、架构层面的建议

### 7.1 线程安全

当前 `GUILogHandler` 使用了线程安全的队列 + 定时轮询机制，这是正确的。但以下场景需要注意：

1. `_on_recorder_state_change` 从录音后台线程调用，直接操作 `_recording_bar.update_state()`。虽然 Qt 的信号-槽机制在跨线程时会自动排队，但直接方法调用不会。建议改为发射信号再连接槽函数。

2. `TranscriptionHandler._poll()` 中通过 `QTimer` 在主线程轮询，这是安全的。但 `_process_message` 中直接读写文件（第 219-222 行）可能阻塞 UI。

### 7.2 错误处理

多处使用了 `except Exception: pass` 或 `except Exception as e: self._log(...)` 的宽泛捕获。建议：
- 区分可恢复错误和致命错误
- 致命错误（如模型加载失败）应弹窗提示而非仅写日志
- 添加用户友好的错误消息

### 7.3 跨平台兼容

项目文档提到需要支持 Mac，但以下代码是 Windows 特定的：
- `subprocess.Popen(["explorer", folder])` — 打开文件夹（仅 Windows）
- PowerShell Toast 通知 — 仅 Windows
- `PyAudioWPatch` — 仅 Windows（WASAPI）

如果未来需要跨平台，这些需要抽象化。

## 九、总结与建议

### 9.1 当前状态评估

侧耳倾听的 PySide6 迁移在**框架层面**已经完成了——所有模块都已用 PySide6 重写，布局系统、事件模型、定时器、多进程通信都做了正确的框架转换。但在**功能连接层面**还有较多遗漏，主要表现为信号/槽未连接和回调未注册。

用项目文档的自评分来看：标注为 80-100% 完成度的模块，实际功能完成度大约在 70-85%。最大的差距在设置页（原版有 12 个配置区域，新版只有 5 个）和部分工具栏按钮的信号连接。

### 9.2 建议实施顺序

1. **第一轮（P0 修复，约 1 小时）**：修复 7 个严重 bug，让核心功能全部可用
2. **第二轮（P1 修复，约 2 小时）**：修复数据流和状态管理问题
3. **第三轮（P2 功能补全，约 1-2 天）**：补全设置页、对话框、录音询问等缺失功能
4. **第四轮（代码质量，约 1 天）**：QSS 样式集中化、解耦、错误处理完善

### 9.3 与原版第三轮 UI 优化的关系

原版 MeetScribe 正在进行第三轮 UI 优化（5 个问题：列头 DPI 错位、滚轮修复、高度自适应、关于区域、API Key 优化）。这些优化是针对 customtkinter 的，不直接适用于侧耳倾听。但其中的**设计决策**可以参考：

- 列头对齐：PySide6 的 `QScrollArea` 不存在 DPI 缩放问题，但需要确保表头和数据列宽度同步
- 滚轮：`QScrollArea` 原生支持滚轮，不需要 `bind_all` 的 workaround
- 高度自适应：Qt 布局系统（QVBoxLayout + stretch factor）原生支持自适应
- 关于区域：新版需要添加版本号和制作者信息（与原版相同的需求）
- API Key 加宽 + 切换按钮：需要在新版中实现
