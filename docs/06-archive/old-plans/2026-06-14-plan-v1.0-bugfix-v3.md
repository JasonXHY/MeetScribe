### BUG-001：录音完成后文件列表不显示新文件

**严重程度**：P0（核心功能不可用）

**现象**：录音停止后，日志显示文件已保存，但主页文件列表中不显示新录音文件。

**日志**：
```
[2026-06-18 11:20:42] INFO - Audio saved: 061811会议.wav (16000Hz, 302.1s)
[2026-06-18 11:20:42] INFO - Audio file saved: C:/侧耳倾听/recordings\061811会议.wav
```

**根因分析**：

录音停止流程：
1. `unified_recorder._stop_and_save()` 在后台线程执行
2. `on_stop_complete(saved_files)` 从后台线程回调
3. `app._on_recorder_stop_complete()` 使用 `QTimer.singleShot(0, ...)` 调度到主线程
4. `app._handle_stop_complete()` → `file_manager.add_file()` → `refresh_file_list()`
5. 紧接着 `ask_transcribe_after_record()` 弹出 QMessageBox

**问题**：`QTimer.singleShot(0, ...)` 的延迟可能不足以确保文件已写入磁盘，或者 `refresh_file_list()` 执行时文件尚未完全可读。

**修复方案**：

在 `_handle_stop_complete` 中，`refresh_file_list()` 之前添加短暂延迟确保文件系统同步：

```python
def _handle_stop_complete(self, saved_files):
    for saved in saved_files:
        self._log(f"录音已保存: {os.path.basename(saved)}")
        # 等待文件系统同步
        import time
        for _ in range(10):
            if os.path.exists(saved) and os.path.getsize(saved) > 0:
                break
            time.sleep(0.1)
        existing = self.file_manager.get_file(saved)
        if not existing:
            self.file_manager.add_file(saved)
    # ... 其余逻辑
```

<!-- [QW-审核] ⚠️ 上述方案有严重问题：
time.sleep(0.1) × 10 = 最多 1 秒阻塞主线程，期间 UI 完全冻结（无法点击、无法刷新）。

建议改用 QTimer 轮询：
```python
def _handle_stop_complete(self, saved_files):
    for saved in saved_files:
        self._log(f"录音已保存: {os.path.basename(saved)}")
        self.file_manager.add_file(saved)
    self.refresh_file_list()
    # 弹窗前确保列表已刷新
    QApplication.processEvents()
```
因为 _handle_stop_complete 本身已经在 QTimer.singleShot(0, ...) 中执行，文件系统写入在 _stop_and_save() 完成后已经同步，不需要额外等待。
真正的问题可能不在文件同步，而在 refresh_file_list() 的执行时序——需要确认 add_file() 是否触发了 _notify() → debounce → refresh 的异步链路。
-->

同时在 `ask_transcribe_after_record` 弹窗关闭后强制刷新：

```python
def ask_transcribe_after_record(self, file_path):
    reply = QMessageBox.question(...)
    if reply == QMessageBox.Yes:
        self._transcribe_single(file_path)
    # 弹窗关闭后强制刷新
    self.refresh_file_list()
```

### BUG-002：转写后主题不写入文件列表

**严重程度**：P0（核心功能不可用）

**现象**：转写完成后，文件列表的"主题"列为空，未显示 AI 摘要提取的主题。

**根因分析**：

转写完成流程：
1. `_on_done()` → `_match_voiceprints()` → `_apply_speaker_names()` → `transcription_done.emit()`
2. `_on_transcription_done_handler()` → `refresh_file_list()`
3. **但** `_generate_summary()` 是异步执行的（通过 `_poll` → `_process_message` → `auto_summary` 消息触发）
4. 摘要生成完成后调用 `update_topic()`，但此时 `refresh_file_list()` 已经执行过了

**问题**：`update_topic()` 在摘要生成时调用，但 `refresh_file_list()` 在 `_on_done` 中先执行，摘要还没生成。

**修复方案**：

在 `_generate_summary` 中，`update_topic()` 之后添加文件列表刷新：

```python
# 提取主题
topic = self._extract_topic_from_summary(summary)
if topic and self._app and hasattr(self._app, 'file_manager'):
    for item in self._app.file_manager.files:
        if item.result_path and os.path.basename(item.result_path).startswith(base_name + "_transcript"):
            self._app.file_manager.update_topic(item.file_path, topic)
            self.log_message.emit(f"会议主题: {topic}")
            break

    # 主题更新后刷新文件列表
    if self._app and hasattr(self._app, '_home_page'):
        QTimer.singleShot(0, self._app._home_page.refresh_file_list)
```

<!-- [QW-审核] 方向正确。但有两点注意：
1. QTimer.singleShot(0, ...) 在 _generate_summary 完成时调用，此时可能已有多个 pending 的 singleShot
2. 更好的方式是直接发射信号：file_manager.file_status_changed → _on_file_status_changed → refresh_file_list()
   这样利用已有的信号链路，不引入新的耦合路径
3. 另外需要确保 _extract_topic_from_summary() 能正确匹配主题格式（v2 问题 C 的根因：只匹配 # 或 ## 开头的行）
-->

### BUG-003：转写内容中发言人未替换为真实姓名

**严重程度**：P0（核心功能不可用）

**现象**：转写结果文件中所有发言人仍显示为 "Speaker 1"、"Speaker 2" 等，未替换为真实姓名。

**但 AI 摘要中显示正确**：摘要中 `### 参会人员` 部分正确显示了 `- [Speaker 1] 万斌` 等。

**根因分析**：

发言人姓名传播链路：
1. AI 摘要生成时，`_extract_speaker_mapping_from_summary()` 从摘要中提取 `Speaker N → 姓名` 映射
2. 调用 `apply_speaker_mapping()` 将姓名写入转写文件
3. **但** `_apply_speaker_names()` 在 `_on_done()` 中调用，此时摘要可能还没生成

**问题**：`_apply_speaker_names()` 使用 SpeakerNamer（正则+LLM）提取姓名，但转写文本中可能没有足够的上下文线索（如"我是XXX"），导致正则提取失败，LLM 兜底也失败。

**而 AI 摘要中的参会人员**是 LLM 在生成摘要时直接识别的，不依赖转写文本中的线索。

**修复方案**：

方案 A（推荐）：在 `_generate_summary` 中，摘要生成后也将 `speaker_mapping` 应用到转写文件：

```python
# 提取参会人员映射
speaker_mapping = self._extract_speaker_mapping_from_summary(summary)
if speaker_mapping and self._app and hasattr(self._app, 'file_manager'):
    for item in self._app.file_manager.files:
        if item.result_path and os.path.basename(item.result_path).startswith(base_name + "_transcript"):
            if os.path.exists(item.result_path):
                apply_speaker_mapping(item.result_path, speaker_mapping)
            if os.path.exists(summary_path):
                apply_speaker_mapping(summary_path, speaker_mapping)
            str_mapping = {str(k): v for k, v in speaker_mapping.items()}
            self._app.file_manager.update_speaker_names(item.file_path, str_mapping)
            break
```

<!-- [QW-审核] ⚠️ 方案 A 会导致双重写入：
_on_done() 中已经调用了 _apply_speaker_names()，如果摘要生成时再调一次 apply_speaker_mapping，
同一个文件会被写两次。如果两次的 mapping 不同（例如 SpeakerNamer 和摘要提取结果不一致），
后写的会覆盖先写的，可能造成混乱。

建议：
1. 保留 _on_done() 中的 SpeakerNamer 提取（快速、不依赖 AI）
2. 在 _generate_summary() 中，如果摘要提取到了更准确的 mapping，**增量覆盖**（只替换仍为 Speaker N 的条目）
3. 用 set 对比两次提取的 mapping，摘要结果优先级更高

另外，还需要确保同时刷新文件列表（与 BUG-002 的修复联动）。
-->

方案 B：调整 `_on_done()` 中的执行顺序，确保摘要生成完成后再执行姓名提取。

### BUG-004：发言人管理弹窗显示"可能是万斌"但实际全错

**严重程度**：P1（功能异常）

**现象**：发言人管理弹窗中，所有说话人都显示"可能是万斌"，但实际转写内容中 Speaker 1/2/3 对应不同的人。

**根因分析**：

发言人管理弹窗中的匹配逻辑：
1. `_add_match_suggestion()` 从声纹嵌入向量匹配音色库
2. 调用 `library.match(embedding)` 返回最相似的人
3. 如果所有说话人的嵌入向量都与"万斌"最相似，就会全部显示"可能是万斌"

**可能原因**：
- 声纹嵌入向量提取有误（所有说话人的嵌入相同或相似）
- 音色库中只有"万斌"一个样本，导致所有匹配都返回"万斌"
- CAM++ 模型提取的嵌入向量质量不佳

<!-- [QW-审核] 根因分析不够深入。根据 v2 的分析，更可能的原因是：
1. **音色库中确实只有一个人**（万斌），所有 embedding 都与他匹配得分最高
2. **嵌入向量为零向量或 NaN**：如果 CAM++ 提取失败，返回零向量，余弦相似度会退化为随机匹配
3. **speaker_id 映射错位**：transcription.py 中 _speaker_embeddings 的 key 是 0-indexed，
   但 dialogs.py 中遍历 speakers 时可能用了不同的 index，导致 embedding 和 speaker 对不上

排查步骤建议：
1. 打印 voiceprint_library.json 中有多少说话人、多少样本
2. 在 _add_match_suggestion() 中打印 score 值（如果全是 0.31 说明匹配质量极低）
3. 检查 embedding 向量是否为非零（np.linalg.norm > 0）
4. 确认 speaker_id 在 dialogs.py 和 transcription.py 中的一致性

修复方案应增加：
- 如果 score < MATCH_THRESHOLD（0.31），不显示建议（目前 match() 已经过滤，但 dialogs 可能没检查）
- 如果音色库为空或只有 1 人，显示"音色库样本不足"提示
-->

**修复方案**：

1. 检查声纹嵌入向量是否正确（`_extract_speaker_embeddings`）
2. 检查音色库中是否有足够的样本
3. 在匹配时添加最低分数阈值，低于阈值不显示建议
