**QW 根因定位（代码 + test2_summary.md 交叉验证）：**

`_extract_topic_from_summary()` 的 `###` 支持存在逻辑错误——它把 section header "会议主题" 本身当成了 topic 文本。

实际 test2_summary.md 内容：
```
### 会议主题
干部管理系统一期使用反馈与二期优化规划讨论
```

逐行跟踪 `_extract_topic_from_summary`：
- 第 1 行 `### 会议主题`：匹配 `startswith("### ")` → `lstrip("# ")` 得到 `"会议主题"` → `len("会议主题") == 4` → **4 > 5 为 False，跳过**
- 第 2 行 `干部管理系统一期使用反馈与二期优化规划讨论`：不以 `#` 开头，不匹配任何分支
- 第 4 行 `### 参会人员`：`lstrip("# ")` → `"参会人员"` → `len == 4` → 跳过
- 其余行均不匹配
- **函数返回 `None`**

所以 `topic` 为 `None`，`update_topic()` 从未被调用，`refresh_needed` 从未发射。

**修复方案**：`###` 开头的行是 section header，不是 topic 内容。应跳过 header 行，取下一行的正文内容作为 topic：

```python
def _extract_topic_from_summary(self, summary):
    try:
        lines = summary.splitlines()
        for i, line in enumerate(lines[:15]):
            line = line.strip()
            # 找到"会议主题"相关的 section header
            if line.startswith("#") and ("主题" in line or "会议" in line):
                # 取下一行非空内容作为实际主题
                for j in range(i + 1, min(i + 4, len(lines))):
                    next_line = lines[j].strip()
                    if next_line and not next_line.startswith("#"):
                        topic = next_line[:50]
                        if len(topic) > 2:
                            return topic
                break
            # 兼容：直接匹配 "主题：xxx" 格式
            if "主题" in line and "：" in line:
                m = re.search(r'主题[：:]\s*(.+)', line)
                if m:
                    topic = m.group(1).strip()
                    if len(topic) > 2:
                        return topic[:50]
    except Exception:
        pass
    return None
```

**附：信号链路确认（均正常）**
- `refresh_needed` 信号已在 `app.py:184` 正确连接 `_home_page.refresh_file_list`
- `update_topic()` 实现正确（`file_manager.py:329`），会调用 `_notify()` + `_save_to_file()`

### 问题 2：CAM++ embedding dim=2（P0，阻塞声纹匹配）

**现象**：所有 8 个 Speaker 的嵌入向量维度为 2（应为 192）。

**日志**：
```
[WORKER] Speaker 0: embedding dim=2
[WORKER] Speaker 1: embedding dim=2
...
[WORKER] Speaker 7: embedding dim=2
```

**影响**：声纹匹配完全失效，所有 Speaker 无法匹配到音色库成员。

**代码位置**：`src/transcriber.py` 第 1109-1132 行 `_extract_per_speaker_embeddings`

```python
result = model_spk.inference(input=segment_audio)
if result and len(result) > 0:
    spk_emb = result[0].get("spk_embedding")  # ← 这里返回的只有 2 个元素
```

<!-- ══════ QW 排查结论 2026-06-18 ══════ -->

**QW 根因定位：CAM++ inference 返回嵌套 batch 维度，代码只剥一层导致 dim 错乱**

`_extract_per_speaker_embeddings()` 中（transcriber.py:1110-1123）：

```python
result = model_spk.inference(input=segment_audio)  # 单段音频
spk_emb = result[0].get("spk_embedding")
# ↓ 当前的维度剥离代码
if len(spk_emb.shape) > 1 and spk_emb.shape[0] == 1:
    spk_emb = spk_emb[0]  # 只剥一层 [1, 192] → [192]
```

**问题分析**：FunASR 的 CAM++ `inference()` 对单段音频的返回格式可能是 `[1, 1, 192]`（batch + extra + embedding）或 `[2, 192]`（双通道误解读）。当前代码只处理 `shape[0] == 1` 的情况，遇到多层嵌套时：

- `[1, 1, 192]` → shape[0]=1 → 剥一层 → `[1, 192]` → 仍嵌套 → 存入 → 后续 `tolist()` 得到 `[[192个浮点数]]` → `len()` = 1
- `[2, 192]` → shape[0]=2 ≠ 1 → 不剥 → 后续 `tolist()` 得到 `[[192], [192]]` → `len()` = 2

`_send_embeddings()` 中打印 `dim=2` 说明是后者——`spk_embedding` 是一个 shape 为 `[2, 192]` 的 tensor，代码没有剥离任何维度，`tolist()` 后 `len(list) == 2`。

**修复方案**：用 `np.squeeze()` 替代手动剥离，自动去掉所有 size=1 的维度：

```python
# 提取嵌入
result = model_spk.inference(input=segment_audio)
if result and len(result) > 0:
    spk_emb = result[0].get("spk_embedding")
    # 兼容不同版本：也可能叫 spk_emb
    if spk_emb is None:
        spk_emb = result[0].get("spk_emb")
    if spk_emb is not None:
        if hasattr(spk_emb, "cpu"):
            spk_emb = spk_emb.cpu().numpy()
        elif hasattr(spk_emb, "tolist"):
            spk_emb = np.array(spk_emb)
        if isinstance(spk_emb, list):
            spk_emb = np.array(spk_emb)
        # ★ 关键修复：用 squeeze 去掉所有 size=1 的维度
        spk_emb = np.squeeze(spk_emb)
        # 校验最终维度（CAM++ 应为 192）
        if spk_emb.ndim != 1:
            logger.warning(f"[SPK-EMB] Unexpected dim after squeeze: {spk_emb.shape}, skipping")
            continue
        spk_embeddings.append(spk_emb)
```

**同时建议在 `_send_embeddings()` 的 dim 日志后加一行验证**：
```python
logger.debug(f"[WORKER] Speaker {spk_id}: embedding dim={emb_len}"
             f"{', WARN expected 192' if emb_len != 192 else ''}")
```

### 问题 3：发言人姓名未自动匹配（P0）

**现象**：转写完成后，发言人管理弹窗中所有说话人显示"可能是万斌"，且转写内容中 Speaker N 未替换为姓名。

**日志**：
```
正则提取完成: 0/6 个说话人已识别
对 6 个未识别说话人启用 LLM 兜底
本地 Ollama 已禁用，跳过说话人姓名提取
LLM 返回结果无效或为空
最终提取完成: 0/6 个说话人已识别
```

<!-- ══════ QW 排查结论 2026-06-18 ══════ -->

**QW 根因定位（test2_summary.md 实证 + 代码跟踪）：**

实际 test2_summary.md 参会人员部分：
```
### 参会人员
[Speaker 1]（项目负责人）
[Speaker 2]（与会人员）
...
[Speaker 6]（与会人员）
```

AI 没有返回真实姓名，而是返回了角色推断。`_extract_speaker_mapping_from_summary()` 的三个正则全部不匹配：

| 正则 | 输入内容 | 匹配结果 | 原因 |
|------|---------|---------|------|
| `\[Speaker\s+(\d+)\]\s*(.+)` | `[Speaker 1]（项目负责人）` | 匹配到，name="（项目负责人）" | 但 `name` 以全角括号开头，后续问题：这不是真实姓名 |
| `Speaker\s+(\d+)[.:：]\s*(.+)` | 同上 | 不匹配 | 没有 `.` `:` `：` 分隔符 |
| `(\d+)[.:：]\s*(.+)"` | 同上 | 不匹配 | 同上 |

**实际跟踪**：第一个正则确实匹配了，`name = "（项目负责人）"`。但这个 name 会被 `apply_speaker_mapping()` 写入转写文件，把 `Speaker 1` 替换为 `（项目负责人）`——这比原来的 Speaker N 更糟。

**两个层面的问题：**

**A. AI 返回角色推断而非姓名**：系统 prompt 允许 AI 在无法确定时使用 `[Speaker N]（角色推断：XXX）` 格式。实际 AI 输出用的是 `[Speaker N]（角色）` 格式（无"角色推断："前缀）。需要在提取逻辑中过滤掉括号包裹的角色描述。

**B. 姓名来源链路断裂**：
1. 正则提取（SpeakerNamer）失败——会议中无人自报姓名 ✓ 符合预期
2. LLM 兜底跳过——Ollama 已禁用 ✓ 符合预期
3. 声纹匹配失败——**问题 2 的 dim=2 bug 导致所有嵌入无效** ← 根本原因
4. AI 摘要参会人员——无真实姓名可用，返回角色推断 ← 符合预期

**修复方案**：

**优先修复问题 2（CAM++ dim=2）**——修复后声纹匹配正常工作，confirmed 匹配会提供真实姓名给 AI 摘要的 prompt，AI 就会输出 `[Speaker N] 万斌` 而非 `[Speaker N]（项目负责人）`。

**同时增强 `_extract_speaker_mapping_from_summary()`**：过滤掉括号包裹的角色描述，避免把"（项目负责人）"当姓名写入：

```python
m = re.match(r'\[Speaker\s+(\d+)\]\s*(.+)', content)
if m:
    spk_num = int(m.group(1))
    name = m.group(2).strip()
    # ★ 过滤角色推断：以全角/半角括号开头的不是姓名
    if name and name.startswith(("（", "(")):
        continue  # 跳过 [Speaker N]（角色推断：XXX）
    if name and len(name) < 20:
        mapping[spk_num] = name
        continue
```

### 问题 4：声纹匹配写入逻辑（P1）

**现象**：多个 Speaker 匹配同一音色库成员时，最高分的误匹配仍被写入，导致 profile 污染。

**日志**：
```
[VOICEPRINT] Skipped conflict: Speaker 3 -> 万斌 (score=0.722, lower than best match)
```

**代码位置**：`src/gui/transcription.py` 第 317-369 行 `_match_voiceprints`

<!-- ══════ QW 排查结论 2026-06-18 ══════ -->

**QW 根因定位：问题 2（dim=2）的直接后果，不是独立 bug**

当所有 8 个 Speaker 的嵌入都是 dim=2 时，余弦相似度完全失去区分能力。2 维空间中随机向量的余弦相似度分布非常集中，导致所有 Speaker 对同一个音色库成员的 score 都差不多，冲突检测自然会跳过大量匹配。

**优先级**：先修问题 2（dim=2），修完后声纹匹配大概率恢复正常。

**但代码层面仍有改进空间**（dim=2 修复后仍建议实施）：

1. **添加 embedding 维度校验**：匹配前检查 dim 是否为预期值（192），异常时直接跳过并报警：
```python
# 在 _match_voiceprints() 中，匹配前校验
embedding = np.array(embedding)
if embedding.shape[0] != 192:
    logger.warning(f"[VOICEPRINT] Speaker {speaker_id+1}: unexpected embedding dim={embedding.shape[0]}, skipping match")
    continue
```

2. **冲突时不自动追加声纹样本**：当前 `_apply_voiceprint_match()` 对 confirmed 级别的匹配自动追加声纹样本（line 413-418）。如果匹配是误匹配（dim=2 导致的假阳性），会污染音色库。建议在存在冲突时不自动追加：
```python
# 只在无冲突时自动追加
has_conflict = any(
    m[1] == name and m[0] != speaker_id for m in all_matches
)
if confidence == "confirmed" and not has_conflict:
    library.add_speaker(name, embedding, ...)
```

3. **音色库中只有 1 人时显示提示**：当前 `voiceprint_library.json` 只有"万斌"一个人的样本，所有 Speaker 都与他匹配。应在 UI 上提示用户添加更多声纹样本。

### 问题 5：Queue poll error（P2）

**现象**：`'NoneType' object has no attribute 'get_nowait'`

**日志**：
```
[2026-06-18 17:03:34] ERROR   MeetScribe - Queue poll error: 'NoneType' object has no attribute 'get_nowait'
```

**代码位置**：`src/gui/transcription.py` 第 186-195 行 `_poll`

<!-- ══════ QW 排查结论 2026-06-18 ══════ -->

**QW 根因定位：`_on_done()` 中 `self._queue = None` 后 poll timer 残留触发**

代码流程：
1. `_poll()` 处理 `"done"` 消息 → 调用 `_on_done()`
2. `_on_done()`: `self._poll_timer.stop()` → 清理进程 → **`self._queue = None`** → `transcription_done.emit()`
3. `transcription_done.emit()` 触发 `_on_transcription_done()` → 弹出 `QMessageBox.information()`
4. **QMessageBox 运行嵌套事件循环**

理论上 timer 已在第 2 步 stop。但在特定时序下（例如 `_on_done()` 执行期间恰好到达 timer tick），Qt 可能已将该 timer event 排入队列。`stop()` 后队列中的 pending event 在某些 Qt 版本中仍可能被 dispatch。

**另一个可能路径**：`_on_done()` → `self._task_queue.complete_current_task()` → `_check_queue()` → `_execute_task()`（如果有排队任务）→ `self._queue = multiprocessing.Queue()` + `self._poll_timer.start()`。新 timer 启动后下一次 `_poll()` 执行时，如果新任务很快完成又调 `_on_done()` → `self._queue = None`，但新 timer 的某个 tick 还挂着。

**修复方案**（防御性编程，成本极低）：

在 `_poll()` 的 `get_nowait()` 前再检查一次 `self._queue`：

```python
def _poll(self):
    """轮询转写进度"""
    if self._queue is None:
        self._poll_timer.stop()
        return

    try:
        while True:
            try:
                # ★ 防御性检查：防止 _on_done() 在循环间隙将 queue 置 None
                if self._queue is None:
                    break
                msg = self._queue.get_nowait()
                self._process_message(msg)
            except queue.Empty:
                break
    except Exception as e:
        logger.error(f"Queue poll error: {e}")
    # ...
```

**P2 优先级合理**，不阻塞当前版本发布。
