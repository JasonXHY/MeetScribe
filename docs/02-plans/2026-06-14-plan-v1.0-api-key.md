## 一、背景与动机

### 1.1 为什么需要预置 API Key

v1.0 是内测版本，目标是让内测用户**开箱即用**，无需自行配置 AI 摘要/纠错的 API Key。

当前问题：
- 用户首次启动后，AI 摘要和纠错功能不可用（无 API Key）
- 内测用户不具备自行申请/配置 API Key 的能力
- 配置流程复杂（需选择厂商、填写 Key、选择接入模式）

### 1.1.1 内测 API 信息

| 项目 | 值 |
|------|-----|
| API Key | `sk-c4wihuvmfe4fv7qurdhk975rinuxflgi056dphuvv25x8lbm` |
| 用途 | 侧耳倾听内测专用通道 |
| 厂商 | 小米 MiMo |
| 模型 | `mimo-v2.5`（标准版，非 Pro） |
| 接入模式 | 按量计费 |
| 有效期 | 限时启用，测试完成后手动停用 |

### 1.2 安全考量

- 预置 Key 为限时启用的测试 Key，过期后手动停用
- **不允许在 UI 中展示或复制**该 Key，防止被滥用
- 用户自己的 Key 可正常显示和管理
- 后台硬编码即可，即使被复制也不影响（限时 Key）
- <!-- [QW] --> **PyInstaller 打包后可被 pyinstxtractor + uncompyle6 反编译提取源码中的 Key**，建议做 base64 混淆增加一层屏障（见 3.1）

### 1.3 设计目标

1. 开箱即用：首次启动即可使用 AI 摘要/纠错
2. 安全：内置 Key 不可查看/复制
3. 可替换：用户可填入自己的 Key 覆盖内置 Key
4. 可恢复：用户可删除自己的 Key，回退到内置 Key
5. 可清理：打包前盘点并移除测试数据

## 二、存储策略

### 2.1 双 Key 架构

| 配置键 | 用途 | 可见性 |
|--------|------|--------|
| `ai_default_api_key` | 内置默认 Key（硬编码） | 不可见 |
| `ai_user_api_key` | 用户自填的 Key | UI 可管理 |

**读取优先级**：`ai_user_api_key` > `ai_default_api_key`

### 2.2 为什么不用单 Key

如果只用一个 `ai_api_key`：
- 内置 Key 会写入 `settings.json`，用户可直接打开文件查看
- 用户填入自己的 Key 后，内置 Key 被覆盖，无法恢复
- 无法区分"用户主动配置"和"使用默认值"

双 Key 方案解决了以上所有问题。

### 3.2 `src/gui/transcription.py` — 支持 fallback + 缓存失效

<!-- [QW] 修改：增加 Key 变更检测和 AIService 缓存失效机制 -->

修改 `_get_ai_service()` 方法：

```python
def _get_ai_service(self):
    # [QW] 新增：检测 Key 是否变化，变了就重建 AIService
    current_key = (
        self._app.config.get("ai_user_api_key", "")
        or self._app.config.get("ai_default_api_key", "")
    )
    if self._ai_service is not None and self._ai_service.api_key != current_key:
        self._ai_service = None

    if self._ai_service is None:
        if not self._app or not hasattr(self._app, 'config'):
            return None

        vendor = self._app.config.get("ai_vendor", "")
        # 优先读用户 Key，为空则读内置 Key
        api_key = self._app.config.get("ai_user_api_key", "")
        if not api_key:
            api_key = self._app.config.get("ai_default_api_key", "")
        if not api_key:
            return None

        self._ai_service = AIService(
            vendor=vendor or "小米 MiMo",
            model=self._app.config.get("ai_model", "mimo-v2.5-pro"),
            access_mode=self._app.config.get("ai_access_mode", "按量计费"),
            api_key=api_key,
        )
    return self._ai_service
```

<!-- [QW] 新增：Key 失效时的用户引导 -->

在调用 `_get_ai_service()` 的地方（如 AI 摘要/纠错），增加错误处理：

```python
def _on_ai_error(self, error_msg):
    """AI 调用失败时的用户引导"""
    user_key = self._app.config.get("ai_user_api_key", "")
    if not user_key:
        # 使用的是内置 Key，可能已过期
        self._show_hint(
            "AI 功能暂不可用（内置 Key 可能已过期），请在「设置」页配置自己的 API Key"
        )
    else:
        self._show_hint(f"AI 功能调用失败：{error_msg}")
```

## 四、打包前盘点

### 4.1 不打包的内容

| 内容 | 原因 | 处理方式 |
|------|------|----------|
| `data/voiceprint_library.json` | 测试数据 | 打包前删除或替换为空库 |
| `data/file_history.json` | 用户数据 | 打包前删除 |
| `recordings/` | 用户录音 | .gitignore + 打包排除 |
| `transcripts/` | 用户转写结果 | .gitignore + 打包排除 |
| `logs/` | 运行日志 | .gitignore + 打包排除 |
| `config/settings.json` | 含用户 API Key | .gitignore + 打包排除 |
| `models_cache/` | AI 模型文件 | 单独分发或首次启动下载 |

### 4.2 打包时路径处理

- 打包路径使用英文 `MeetScribe`，避免中文路径问题
- 配置文件路径使用相对路径或 `--add-data` 打包默认配置
- 内置 API Key 硬编码在 `config.py` 源码中，随代码打包

### 4.3 首次启动流程

```
启动 → 检测 settings.json 不存在
  → 使用 DEFAULTS（含内置 API Key）
  → AI 摘要/纠错开箱即用
  → 用户可在设置页填入自己的 Key
```

## 六、注意事项

1. **内置 Key 时效性**：内置 Key 为限时测试 Key，过期后需手动停用或更新
2. **Key 安全**：即使源码被反编译获取到 Key，也因限时而影响有限
3. **测试数据清理**：打包前必须清理 `data/` 目录下的测试数据
4. **模型分发**：AI 模型（~2GB）不随代码打包，首次启动时引导下载

<!-- [QW] 新增以下注意事项 -->

5. **共享 Key 限额风险**：所有内测用户共用一个内置 Key，需关注厂商的日限额和并发限制。建议选择限额较高的免费模型（如智谱 GLM-4.7-Flash、百川 M3 Plus），并在内测群监控用量。一旦触发限额，所有用户同时不可用。
6. **Key 过期后的用户体验**：内置 Key 过期后，用户点 AI 摘要/纠错会报错。需确保错误提示明确引导用户去设置页配置自己的 Key（见 3.2 的 `_on_ai_error`），而不是显示模糊的 API 认证失败。
7. **Key 轮换策略**：如果需要更换内置 Key，当前方案需要重新编译和分发。如需远程更新，可考虑将默认 Key 放在启动时拉取的远程配置中（v1.1 考虑）。
8. **PyInstaller 反编译防护**：base64 混淆仅是最低限度防护，专业用户仍可提取。对于内测版本足够，正式版应考虑服务端代理方案。
